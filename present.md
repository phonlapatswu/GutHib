### Slide 1: Title Slide
**บทพูด:** "สวัสดีครับอาจารย์ และเพื่อนๆ ทุกคน วันนี้ผมจะมานำเสนอโปรเจกต์ Shark Task แพลตฟอร์มบริหารจัดการงานยุคใหม่ที่ถูกพัฒนาขึ้นภายใต้แนวคิด 'Project Management with Teeth' หรือการจัดการที่มีประสิทธิภาพ เฉียบคม และรวดเร็วครับ..."

### Slide 2: 1. SDLC: Iterative & Incremental Model
**บทพูด:** "ในด้านกระบวนการทำงาน เราเลือกใช้โมเดล Iterative & Incremental ครับ เราเริ่มต้นจากการทำตัวต้นแบบในฟีเจอร์หลัก (Core) อย่างระบบสมาชิกและฐานข้อมูลก่อน จากนั้นจึงหมุนรอบการทำงานเพิ่มฟีเจอร์ที่ซับซ้อนขึ้นอย่าง Gantt Chart และระบบแชลสดทีละส่วน ทำให้เราสามารถตรวจสอบคุณภาพและแก้ไขได้ทันทีในแต่ละ Sprint ครับ..." 
**💡 เคล็ดลับ:** เน้นย้ำว่าเรา 'วัดผลจริง' ในทุกรอบการพัฒนา

---

### Slide 3: 2. Architecture Design Patterns
**บทพูด:** "โครงสร้างระบบของเราเน้นหลักการ Separation of Concerns ครับ โดยแยกส่วน Frontend (Next.js) ออกจาก Backend (Express) อย่างชัดเจน มีการใช้ Middleware ในการคุมเรื่องสิทธิ์และความปลอดภัย และใช้ Axios Interceptor ในการจัดการ Token แบบจุดเดียว เพื่อให้โค้ดสะอาดและดูแลรักษาง่ายที่สุดครับ..." 

---

### Slide 4: 3. Data Architecture & Relational Mapping
**บทพูด:** "มาดูส่วนของข้อมูลครับ เราใช้ PostgreSQL ร่วมกับ Prisma ORM มีการทำ Normalization ข้อมูลอย่างถูกต้อง มีการใช้ Junction Table อย่าง ProjectsOnUsers เพื่อรองรับความสัมพันธ์แบบ Many-to-Many ที่ซับซ้อน และที่สำคัญคือผลจากการตรวจประสิทธิภาพ เราได้เพิ่ม B-Tree Index ในฟิลด์วันที่เพื่อเพิ่มความเร็วในการ Query ข้อมูลจำนวนมากครับ..." 

---

### Slide 5: 4. Technical Depth: Gantt Timeline Logic
**บทพูด:** "หัวใจหลักด้าน Visual ของเราคือ Gantt Chart ครับ เราไม่ได้ใช้ Library สำเร็จรูปที่หนักเครื่อง แต่เราเขียน Coordinate System ขึ้นมาเองโดยใช้สูตรการคำนวณตามที่เห็นในสไลด์ครับ คือการหาความต่างของวันและแปลงเป็น Column ใน CSS Grid ทำให้การแสดงผลรวดเร็วและแม่นยำระดับพิกเซลครับ..." 
**💡 เคล็ดลับ:** ถ้าอาจารย์ถามเรื่องประสิทธิภาพ ให้ตอบว่าเราลดขนาดลงได้เยอะเพราะเขียนเอง

---

### Slide 6: 5. QA Strategy: The Multi-tier testing
**บทพูด:** "คุณภาพซอฟต์แวร์เป็นเรื่องที่เราให้ความสำคัญที่สุดครับ เราทำ Testing ครบทุกระดับ ตั้งแต่ Unit Test ตรรกะใน Backend, API Test ตรวจสอบ Endpoint จนถึง UI Test จำลองการใช้งานจริงผ่าน Browser ซึ่งผลลัพธ์คือเรามี Statement Coverage สูงถึง 83.6% ครอบคลุม Business Logic ทั้งหมดครับ..." 

---

### Slide 7: 6. DevOps: Parallel CI/CD Pipeline
**บทพูด:** "ในส่วนของ DevOps เราใช้ GitHub Actions สร้าง Pipeline อัตโนมัติครับ จุดเด่นคือเราใช้เทคนิค Parallelization รันเทสต์ Backend และ Frontend ไปพร้อมๆ กัน ทำให้ประหยัดเวลาในการ Build ลงถึง 50% ช่วยให้กระบวนการ Continuous Integration ของทีมทำได้รวดเร็วขึ้นมหาศาลครับ..." 

---

### Slide 8: 7. Profiling: Measure -> Analyze -> Fix
**บทพูด:** "ก่อนที่เราจะจบโปรเจกต์ เราได้ทำการ Profiling ระบบครับ โดยใช้หลักการ 'วัดผล-วิเคราะห์-แก้ไข' เราทำทั้ง Static Audit เพื่อหาช่องโหว่ความปลอดภัย และ Dynamic Audit เพื่อวัดขนาดไฟล์และเวลาที่ใช้ในการ Compile จริง เพื่อนำข้อมูลที่ได้มาจูนระบบต่อครับ..." 

---

### Slide 9: 8. Optimization: Technical Rationales
**บทพูด:** "จากผล Profiling เราได้ทำการ Optimization 4 จุดสำคัญครับ: หนึ่งคือการอัปเกรด Next.js เพื่ออุดช่องโหว่ SSRF, สองคือการทำ Code Splitting เพื่อให้หน้าเว็บโหลดเฉพาะสิ่งที่จำเป็น, สามคือ Image Optimization และสี่คือการทำ DB Indexing ซึ่งทั้งหมดนี้ทำโดยมีเป้าหมายด้านประสิทธิภาพที่ชัดเจนครับ..."

### Slide 10: 9. Verification: Final Success Metrics
**บทพูด:** "ผลลัพธ์ที่ได้จากการจูนระบบนั้นชัดเจนมากครับ เราสามารถปิดช่องโหว่ความปลอดภัยได้ 100%, ลดขนาดไฟล์หน้าหลักลงได้ถึงครึ่งหนึ่ง (จาก 13.4 เหลือ 6.6 KB) และลดเวลาในการ Build ลงได้ 14% ทำให้ระบบ Shark Task ของเราทั้งเร็วและปลอดภัยตามมาตรฐาน SE ครับ..." 
**💡 เคล็ดลับ:** โชว์ตัวเลขเหล่านี้ด้วยความมั่นใจ เพราะนี่คือ 'ผลงาน' ของคุณ

### Slide 11: 10. Risk Management & Backlog

## 🏆 คำแนะนำเพิ่มเติมสำหรับการพรีเซนต์
1.  **การตอบคำถามอาจารย์**: หากอาจารย์ถามเรื่อง 'ทำไมไม่แก้ Bug ในสไลด์ 11 เลย?' ให้ตอบว่า 'เรายึดหลัก Agile ที่ให้ความสำคัญกับการปิดฟีเจอร์หลัก (Product Vision) และความปลอดภัยของระบบ (Optimization) ก่อน ส่วน Bug เหล่านั้นถูกจัดความสำคัญ (Prioritize) ให้อยู่ใน Sprint ถัดไปตามแผน Risk Management ครับ'
2.  **ความมั่นใจ**: ในสไลด์ที่โชว์ตัวเลข (Coverage 83.6% หรือ JS 6.6 KB) ให้เน้นเสียงในส่วนนั้น เพราะเป็นตัวบ่งบอกว่าคุณเข้าใจระบบเชิงลึกจริงๆ

## 🐙 การพรีเซนต์ Git & GitHub
หากอาจารย์ขอดูประวัติการทำงานใน Git ให้เปิดหน้า GitHub และเน้นจุดต่างๆ ดังนี้ครับ:
1.  **Atomic Commits**: ชี้ให้เห็นว่าเรามีการ Commit แยกส่วนชัดเจน (เช่น `docs:`, `refactor:`, `feat:`) ไม่ได้แก้ทุกอย่างรวมกัน
2.  **Commit Frequency**: แสดงให้เห็นความสม่ำเสมอในการพัฒนา (เรามีการเก็บประวัติการปรับจูนโค้ดอย่างต่อเนื่อง)
3.  **Code Maintenance**: โชว์ว่าเรามีการกลับมาเพิ่ม JSDoc และ Refactor โค้ดให้สะอาดหลังจากการทดสอบ ซึ่งเป็นสิ่งที่วิศวกรซอฟต์แวร์ที่ดีต้องทำครับ

### 🧪 สไลด์: Quality Assurance & Testing (การประกันคุณภาพซอฟต์แวร์)
*(แสดงหน้าจอ Terminal และเปิดไฟล์ TESTING.md)*

**บทพูด:**
"สุดท้ายที่ขาดไม่ได้ในโปรเจกต์วิศวกรรมซอฟต์แวร์คือการทำ **Testing** ครับ ใน Shark Task เราเน้นความน่าเชื่อถือของระบบด้วยการเขียน **Integration Tests** ทั้งหมด **29 Test Cases** ครับ"

"เราใช้ **Jest** และ **Supertest** ในการจำลองการทำงานของ API โดยเรามีการทำ **Mocking Database** เพื่อให้การทดสอบรันได้เร็วและไม่กระทบกับข้อมูลจริงครับ"

"จุดเด่นที่เราทดสอบคือ:
1. **RBAC (Role-Based Access Control):** ตรวจสอบว่าสิทธิ์ Manager เท่านั้นที่สร้างโปรเจกต์ได้ ส่วน Worker จะถูกปฏิเสธ (403)
2. **State Logic:** ตรวจสอบว่าเมื่อเปลี่ยนสถานะเป็น 'In Progress' ระบบจะบันทึกเวลาเริ่มงานให้อัตโนมัติจริงไหม
3. **Validation:** ตรวจสอบความถูกต้องของข้อมูล เช่น รูปแบบอีเมล และความยาวรหัสผ่านครับ"

*(โชว์การรันคำสั่ง `npm test` ใน Terminal)*
"เดี๋ยวผมจะรัน Test ให้ดูสดๆ เลยครับ... ทุกอย่าง Pass 100% ครับ ซึ่งเป็นการยืนยันว่าฟีเจอร์ที่เราทำมาทำงานได้ถูกต้องตาม Logic ที่วางไว้ครับ"

### ⚡ สไลด์: Performance & Optimization (การวิเคราะห์ประสิทธิภาพ)
*(โชว์ไฟล์ PROFILING.md และหน้าจอ Build Report)*

**บทพูด:**
"นอกจากความถูกต้องแล้ว เรายังยึดหลัก **Performance as a Feature** ครับ ผมได้ทำการทำ **Profiling** ระบบใน 2 มิติ คือ Static และ Dynamic เพื่อพิสูจน์ความเร็วครับ"

"ในเชิง **Static**: เราวิเคราะห์ Bundle Size พบว่าไฟล์หลักของเรามีขนาดเพียง 87 KB เท่านั้น ซึ่งถือว่าเบามาก ทำให้ผู้ใช้เปิดหน้าเว็บได้แทบจะทันที (Hydration Fast) ครับ"

"ในเชิง **Dynamic**: เราตรวจสอบผ่าน **Flame Graph** ใน Chrome DevTools เพื่อดูการ Render ของ Gantt Chart ที่ซับซ้อน พบว่าด้วยการใช้ `useMemo` และคัดกรองการ Render เฉพาะจุด ทำให้ระบบรักษาความลื่นไหลได้ที่ 60 FPS ตลอดเวลาครับ"

"สรุปได้ว่า Shark Task ไม่ได้แค่ 'ใช้ได้' แต่ 'ใช้ดี' และพร้อมรองรับการขยายตัว (Scalability) ในอนาคตด้วยครับ"

*(เปิดหน้า GitHub แท็บ Actions และแสดงไฟล์ CICD.md)*

**บทพูด:**

"ทุกครั้งที่มีการเปลี่ยนโค้ด ระบบหุ่นยนต์เบื้องหลังจะทำการ 'ตรวจสอบตัวเอง' อัตโนมัติใน 2 ส่วนสำคัญ:
1. **Automated Testing:** รัน Integration Tests ทั้งหมดเพื่อเช็คความปลอดภัย
2. **Build Validation:** เช็คว่าโค้ดหน้าบ้านยังสร้างออกมาเป็นไฟล์ที่สมบูรณ์ได้จริงไหม"

"หากมีส่วนไหนพัง ระบบจะแจ้งเตือนเราทันที และไม่อนุญาตให้เอาโค้ดนั้นขึ้นระบบจริงครับ ซึ่งนี่คือมาตรฐานที่ใช้กันในบริษัท Tech ชั้นนำทั่วโลก เพื่อให้มั่นใจว่าซอฟต์แวร์ของเรา 'Zero Bug' และมีความเสถียรสูงสุดตลอดเวลาครับ"

"และทั้งหมดนี้คือ Shark Task — GutHib สำหรับคนทำงานทุกคนครับ ขอบคุณครับ"
